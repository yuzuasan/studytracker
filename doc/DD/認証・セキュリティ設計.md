# ■ 認証・セキュリティ設計

---

## 1. JWT設計

### ■ トークン構造（Payload）

JWTのペイロードには以下の情報を含める。

| フィールド | 型      | 内容                |
| ----- | ------ | ----------------- |
| sub   | String | ユーザー識別子（userId）   |
| exp   | Long   | 有効期限（UNIXタイムスタンプ） |
| iat   | Long   | 発行時刻              |

---

### ■ 例（Payload）

```json
{
  "sub": "1",
  "iat": 1710000000,
  "exp": 1710003600
}
```

---

### ■ 有効期限

* アクセストークン：**1時間（3600秒）**

#### ■ 設計理由

* セキュリティと利便性のバランスを考慮
* 個人開発用途のため短すぎる期限は採用しない

---

### ■ リフレッシュトークン

👉 **本システムでは採用しない**

#### ■ 理由

* 学習用途のため構成をシンプルにする
* トークン更新処理の複雑化を避ける
* 再ログインで対応可能

---

### ■ 署名アルゴリズム

* HS256（HMAC + SHA-256）

---

### ■ シークレットキー

* サーバ側で安全に管理
* ソースコードにハードコードしない（環境変数で管理）

---

---

## 2. パスワード管理

### ■ ハッシュアルゴリズム

* **BCrypt**

---

### ■ 設計内容

| 項目     | 内容                              |
| ------ | ------------------------------- |
| アルゴリズム | BCrypt                          |
| エンコード  | Spring Security PasswordEncoder |
| 平文保存   | 不可                              |

---

### ■ ソルト

👉 **BCryptにより自動付与**

#### ■ 補足

* 明示的なソルト管理は不要
* ハッシュごとに異なるソルトが付与される

## 3. 認可制御（Authorization）

---

### ■ 基本方針

👉 **ユーザーは自分のデータのみ操作可能**

---

### ■ 対象リソース

* study_records
* tags
* goals

---

### ■ 制御方法

#### ■ Repositoryレベルで制限

```id="4e8hvp"
findByIdAndUserId(id, userId)
```

👉 他ユーザーのデータを取得できない構造にする

---

### ■ エラー

| ケース      | ステータス                    |
| -------- | ------------------------ |
| 他人のデータ操作 | 404 Not Found |

## 4. 認証フィルタ設計

---

### ■ 構成

Spring Securityを使用し、JWT認証フィルタを実装する。

---

### ■ 処理フロー

1. リクエスト受信
2. Authorizationヘッダを取得
3. JWTを検証
4. userIdを取得
5. SecurityContextにセット

---

### ■ ヘッダ形式

```id="rcq2sw"
Authorization: Bearer {JWT}
```

---

### ■ フィルタ処理（概要）

```id="4l7v0x"
1. トークン取得
2. 署名検証
3. 有効期限チェック
4. userId抽出
5. 認証情報セット
```

---

### ■ 認証失敗時

| ケース    | ステータス            |
| ------ | ---------------- |
| トークンなし | 401 Unauthorized |
| トークン不正 | 401 Unauthorized |
| 有効期限切れ | 401 Unauthorized |
