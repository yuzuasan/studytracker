JWT（JSON Web Token）は、WebアプリやAPIでよく使われる「ユーザーの認証・認可のためのトークン形式」です。特にログイン状態の管理や、サーバー間で安全に情報をやり取りするために使われます。

---

## ■ JWTの基本構造

JWTは「`.`（ドット）」で区切られた3つの部分から構成されています：

```
xxxxx.yyyyy.zzzzz
```

### ① ヘッダー（Header）

* トークンの種類（JWT）や署名アルゴリズム（例：HS256, RS256）を記述

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

### ② ペイロード（Payload）

* ユーザー情報や属性（クレーム）を含む

```json
{
  "sub": "1234567890",
  "name": "Taro",
  "exp": 1710000000
}
```

* よくあるクレーム：

  * `sub`：ユーザーID
  * `exp`：有効期限
  * `iat`：発行時間

※ ペイロードは**暗号化ではなくエンコード**されているだけなので、誰でもデコード可能です。

### ③ 署名（Signature）

* ヘッダーとペイロードを元に、秘密鍵で署名
* 改ざん検知に使われる

---

## ■ JWTの仕組み（流れ）

1. ユーザーがログイン
2. サーバーがJWTを発行
3. クライアント（ブラウザなど）がJWTを保存（例：ローカルストレージ）
4. リクエスト時にJWTを送信（通常はAuthorizationヘッダー）
5. サーバーが署名を検証して認証

---

## ■ メリット

* **ステートレス**（サーバーがセッションを保持しない）
* スケーラビリティが高い（分散システムに向いている）
* 異なるサービス間で認証情報を共有しやすい

---

## ■ デメリット・注意点

* トークンが漏洩すると不正利用される
* 一度発行すると失効させにくい（ブラックリスト管理が必要）
* ペイロードに機密情報を入れてはいけない

---

## ■ よくある用途

* API認証（Bearerトークン）
* シングルサインオン（SSO）
* モバイルアプリの認証

---

## ■ セッション認証との違い

| 項目   | JWT    | セッション  |
| ---- | ------ | ------ |
| 状態管理 | ステートレス | ステートフル |
| 保存場所 | クライアント | サーバー   |
| スケール | 強い     | 弱い     |

---

## ■ まとめ

JWTは「軽量で自己完結型の認証トークン」で、現代のWeb APIやマイクロサービスで広く使われています。ただし、セキュリティ設計（有効期限・HTTPS・保存方法）が非常に重要です。
