# ■ ビジネスロジック詳細設計

---

## ■ 1. 設計方針

---

### ■ レイヤ責務

| レイヤ        | 責務            |
| ---------- | ------------- |
| Controller | リクエスト受付・DTO変換 |
| Service    | ビジネスロジック      |
| Repository | DBアクセス        |

---

### ■ 原則

* 認可はRepositoryで担保（findByIdAndUserId）
* Serviceにロジックを集約
* Controllerは薄く保つ

---

---

# ■ 2. 認証ロジック

---

## ■ 2.1 ユーザー登録

---

### ■ 処理フロー

```
1. リクエストバリデーション
2. username重複チェック
3. passwordをBCryptでハッシュ化
4. User生成
5. createdAt / updatedAt設定
6. DB保存
7. レスポンス返却
```

---

### ■ ビジネスルール

| 項目             | 内容      |
| -------------- | ------- |
| username       | 一意      |
| password       | ハッシュ化必須 |
| githubUsername | 任意      |

---

### ■ エラー

| 条件         | 内容  |
| ---------- | --- |
| バリデーションエラー | 400 |
| username重複 | 409 |

---

---

## ■ 2.2 ログイン

---

### ■ 処理フロー

```
1. リクエストバリデーション
2. usernameでユーザー取得
3. パスワード照合（BCrypt）
4. JWT生成
5. レスポンス返却
```

---

### ■ ビジネスルール

* 認証失敗時は詳細を返さない

---

### ■ エラー

| 条件         | 内容  |
| ---------- | --- |
| バリデーションエラー | 400 |
| ユーザーなし     | 401 |
| パスワード不一致   | 401 |

---

---

## ■ 2.3 認証情報取得（共通）

---

### ■ 方法

* JWTからuserIdを取得
* SecurityContextに保持

---

### ■ Service利用方法

```java
Long userId = currentUserProvider.getUserId();
```

# ■ 3. ユーザーAPI

---

## ■ 3.1 自分の情報取得

### GET /users/me

---

### ■ 処理フロー

```
1. 認証情報からuserId取得
2. userIdでユーザー取得
3. DTO変換
4. レスポンス返却
```

---

### ■ ビジネスルール

* 認証必須
* 自分の情報のみ取得可能

---

### ■ エラー

| 条件              | 内容  |
| --------------- | --- |
| 未認証             | 401 |
| ユーザーなし（通常発生しない） | 404 |

---

---

# ■ 4. 学習記録ロジック

---

## ■ 4.1 学習記録登録

---

### ■ 処理フロー

```
1. リクエストバリデーション
2. ログインユーザー取得
3. Entity生成
4. createdAt / updatedAt設定
5. 保存
6. レスポンス返却
```

---

### ■ ビジネスルール

| 項目           | 内容          |
| ------------ | ----------- |
| user_id      | ログインユーザー固定  |
| studyMinutes | 1〜1440      |
| date         | 過去・未来どちらも許容 |

---

### ■ エラー

| 条件         | 内容  |
| ---------- | --- |
| バリデーションエラー | 400 |
| 未認証        | 401 |

---

---

## ■ 4.2 一覧取得

---

### ■ 処理フロー

```
1. 認証情報からuserId取得
2. userIdで検索
3. 学習日降順で取得
4. DTO変換
5. レスポンス返却
```

---

### ■ ビジネスルール

* 他ユーザーのデータは取得不可
* ページングは未対応（Phase2）

---

### ■ エラー

| 条件  | 内容  |
| --- | --- |
| 未認証 | 401 |

---

---

## ■ 4.3 詳細取得

---

### ■ 処理フロー

```
1. 認証情報からuserId取得
2. findByIdAndUserId
3. DTO変換
4. レスポンス返却
```

---

### ■ エラー

| 条件    | 内容  |
| ----- | --- |
| 未認証   | 401 |
| データなし | 404 |

---

---

## ■ 4.4 更新（PATCH）

---

### ■ 処理フロー

```
1. リクエストバリデーション
2. 認証情報からuserId取得
3. findByIdAndUserId
4. 更新対象項目の判定（nullチェック）
5. nullでない項目のみ更新
6. updatedAt更新
7. 保存
8. レスポンス返却
```

---

### ■ 更新ルール

| 項目      | 内容    |
| ------- | ----- |
| null項目  | 更新しない |
| 全項目null | 400   |

---

### ■ エラー

| 条件         | 内容  |
| ---------- | --- |
| バリデーションエラー | 400 |
| 未認証        | 401 |
| データなし      | 404 |

---

---
